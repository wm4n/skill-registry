# 知識抽取指引

本文件是 `learn-from-repo` skill 的知識抽取權威參考，定義從 PR 內容判斷「哪些內容值得寫入知識庫」的判準、候選知識的呈現格式，以及去重與衝突處理規則。三種知識類型（`business-logic`、`architecture`、`lessons-learnt`）與知識條目範本定義於 `knowledge-format.md` 章節 1 與章節 5，本文所有輸出皆須符合該範本。

---

## 章節 1：三類知識的判準

抽取知識前，先判斷內容屬於哪一類目錄。每一類都有明確的定義與正反例，用來排除不該記錄的雜訊。

### business-logic（商務邏輯）

**定義**：領域規則、業務約束、計算邏輯背後的商業原因。重點是「為什麼系統要這樣運作」，而非程式碼本身如何實作。

- 正例：「訂單 30 分鐘未付款自動取消（防超賣）」
- 正例：「VIP 折扣不可與優惠券疊加（JIRA 單載明財務要求）」
- 反例：「修正 typo」
- 反例：「升級套件版本」

### architecture（架構決策）

**定義**：技術選型、模式採用、邊界劃分的 what & why。重點是決策本身與其背後的取捨理由，而非單純的程式碼調整。

- 正例：「改用 outbox pattern 確保事件不遺失（曾因直接發送遺失事件）」
- 正例：「拆出 rate-limiter middleware 供多服務共用」
- 反例：純重構無決策意涵
- 反例：格式化調整

### lessons-learnt（教訓）

**定義**：曾造成 bug、事故或 rework 的原因與預防方式。判斷訊號包含：PR 是 hotfix／revert／二次修正、review comments 中出現「上次就是這裡出問題」類討論、Jira 單是 Bug 型且描述了根因。

- 正例：「批次任務未設 DB timeout 導致連線池耗盡」
- 正例：「排程任務以本機時區計算執行時間，日光節約時間切換日重複執行，緊急 revert 後二次修正——排程時間一律以 UTC 儲存與計算」
- 反例：一次性的環境問題
- 反例：無普遍性的個案

## 章節 2：值得記 vs 不值得記

判斷完類型後，還需要判斷這筆內容是否真的值得寫入知識庫。核心測試只有一個：

> **三個月後的新成員（或 AI）讀到這條，會做出不同的、更好的決定嗎？**

若答案是否定的，就不該產出這筆知識。以下三種情況通常不值得記：

- 可直接從程式碼看出的事實
- 純機械性變更（rename、bump version、lint fix）
- 只描述「改了什麼」而說不出「為什麼」的變更

一個 PR 完全可能經過完整檢視後，仍然沒有任何內容通過測試。**一個 PR 可能產出 0 筆知識——寧缺勿濫，0 筆是正常結果。** 不應為了湊數而強行產出低品質或無普遍性的條目。

## 章節 3：候選知識呈現格式

抽取出的候選知識在寫入知識庫前，先以下列格式呈現給使用者做人工確認：

```markdown
### 候選 N（新增｜更新｜衝突）
- **類型/模組**：lessons-learnt / payment
- **標題**：批次任務必須設定 DB connection timeout
- **摘要**：一句話
- **來源**：PR #78、JIRA-456（2026-04-22）
- **完整內容**：（依 knowledge-format.md 範本填好的完整條目）
- **若為更新/衝突**：指出既有條目位置（檔案 + 標題）與差異說明
```

「完整內容」欄位必須是已依 `knowledge-format.md` 章節 5 範本（5.1 商務邏輯／架構決策範本或 5.2 Lessons-learnt 範本）填寫完成、可直接寫入知識庫檔案的條目全文，而非草稿或摘要。

## 章節 4：去重與衝突

為避免知識庫內容重複或前後矛盾，抽取候選知識前必須先讀取目標模組的現有知識檔與 `INDEX.md`，再依下列規則分類：

1. **語意重複**：新內容與既有條目描述同一規則，只是措辭不同 → 標記為「更新」，合併進既有條目。合併時遵循 `knowledge-format.md` 章節 5.3 的更新規則：直接在原條目**改寫**內文，並**追加**新的來源（保留舊來源）。
2. **衝突**：新變更推翻既有知識的內容（例如「30 分鐘改為 15 分鐘」）→ 標記為「衝突」，交由人工確認時決定處理方式：預設做法是改寫（等同「更新」，套用同一套改寫並追加來源的規則），或選擇保留歷史註記（在條目中同時保留新舊內容並註明取代關係）。
3. **同批次內合併**：同一批次處理的多個 PR 若觸及同一筆知識，先在批次內部合併為單一候選，再呈現給使用者確認，避免同一條知識產生多筆重複候選。

---

## 章節 5：來源標注

每筆候選知識都必須附上完整的來源資訊，供日後追溯：

- PR 編號 + URL
- Jira 單號（若有）
- merge 日期

當 Jira 資訊撈取失敗時，不應中斷該筆知識的產出，而是在來源欄位中標注「（Jira 脈絡缺失：KEY-123 無法取得）」，知識內容則基於 PR 本身的文字內容與程式碼產出。
